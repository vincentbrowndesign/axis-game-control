# AXIS MD MANAGEMENT

## Source Of Truth

Axis is infrastructure disguised as a cool AI memory app for basketball reality.

Axis is memory-first, not livestream-first.

The product should feel beautiful, calm, cinematic, conversational, alive, fluid,
modern, intelligent, and immersive.

Underneath, Axis remains deeply structured:

- Memory graph.
- Rail-first interaction.
- Contextual overlays.
- Replay infrastructure.
- Continuity engine.
- Semantic retrieval.
- Contextual analytics.
- CV orchestration.
- State-dependent memory reasoning.
- Persistent environment logic.

The user should mostly feel the magic, not the machinery.

The camera is content inside the shell. It is not the application itself.

## Forward Gravity

Every refinement must make Axis feel lighter, cooler, faster, smarter, and more
immersive as the infrastructure deepens.

Do not slow the product down with over-architecture, over-explanation, page-heavy
logic, visible systems thinking, or infrastructure-first aesthetics.

The deeper the system becomes, the simpler and more natural the surface should
feel.

## Locked Structure

Preserve these foundations:

- Shell architecture.
- Edge-to-edge content region.
- Contextual overlays.
- Universal memory rail.
- Memory graph and state graph.
- Upload, replay, auth, Supabase, storage, voice, and AI utility infrastructure.
- Persistent environment logic.

Do not restart the visual language. Evolve it forward.

## Native Camera Direction

LIVE means active memory observation.

It does not mean livestream production.

Replace any "web video embedded in a webpage" assumption with native camera
content inside a persistent shell.

The shell remains stable and owns:

- Scoreboard.
- Telemetry.
- Memory rail.
- Session state.
- Retrieval state.
- Continuity state.
- AI orchestration state.

The camera exists inside the content region only. It captures the world state
while Axis records memory, replay, chronology, continuity, and retrieval anchors.

The camera should feel immersive, stable, native, cinematic, operational, and
uninterrupted.

Reference behavior:

- Instagram camera.
- TikTok camera.
- Snapchat camera.
- Apple Camera.
- Blackmagic camera systems.

Avoid webcam webpage behavior, browser video energy, OBS-style UI, floating
controls on video, livestream dashboard feeling, and streaming-tool complexity.

The camera is world content. The shell frames the world. The world does not
control the shell.

The native camera direction should protect viewport integrity: no Safari zoom
drift, keyboard instability, shell shifting, browser reflow, overlay collisions,
or webpage feeling.

## Surface Direction

Axis should feel closer to modern AI-native consumer systems than sports
management software.

Reference energy:

- ChatGPT.
- Arc.
- Perplexity.
- Apple Sports.
- Twitch.
- TikTok Live.
- Modern AI-native shells.

Avoid enterprise dashboards, admin panels, tactical software, rigid sports
software patterns, obvious AI engineering visuals, and visible database
architecture.

Consumer-beautiful on the surface. Infrastructure-powerful underneath.

## Memory Rail

The rail is the emotional center of Axis.

It should feel alive, permissive, and natural. The user can speak, ask, retrieve,
replay, stat, inspect, and navigate memory through one fluid interaction surface.

The rail is not command syntax. It is contextual basketball language.

During LIVE, the rail orchestrates memory and intelligence around the camera.
The user should feel they are naturally interacting with the live world while
Axis quietly remembers everything.

Keep:

- Enter submits.
- Focus remains active.
- Keyboard remains open.
- Duplicate submit mechanics stay out of the operator flow.
- Basketball phrasing stays natural.

## Content Region

The world stays primary.

The content region should remain cinematic, immersive, edge-to-edge, emotionally
calm, replay-first, and visually breathable.

Do not crowd it with panels, widgets, permanent stat blocks, clip grids, or
software explanations.

Camera content belongs here as the live world layer, not as shell structure.

## Overlays

Overlays remain contextual, intelligent, adaptive, translucent, low-noise, and
temporary.

Reveal intelligence only when it helps the moment.

Never expose backend reasoning chains, orchestration panels, model plumbing, or
system diagrams as product UI.

## Design Language

Increase:

- Premium restraint.
- Environmental depth.
- Compositional confidence.
- Conversational openness.
- Cinematic continuity.
- Native shell integrity.
- Immersive interaction.
- Adaptive intelligence surfaces.
- Low visual entropy.

Reduce:

- Visible mechanics.
- Rigid system framing.
- Software heaviness.
- Engineering exposure.
- Page fragmentation.
- Feature clutter.
- Excessive operational exposition.
- Webcam webpage behavior.
- Browser video energy.
- Livestream tool feeling.

## Drift To Remove

Remove or rewrite anything that creates:

- Dashboard energy.
- Webpage feeling.
- Over-architected UX.
- Rigid feature separation.
- Page-heavy product logic.
- Visible systems thinking.
- Over-described AI behavior.
- Infrastructure-first aesthetics.
- Camera-as-application structure.
- OBS-style UI.
- Streaming-tool complexity.
- Giant settings structures.
- Manual clipping or timeline editing surfaces.
- AI assistant chat UI.
- Scouting analytics or basketball IQ scores.

## Operating Rule

Keep the infrastructure strong and hidden.

Keep the interaction natural.

Keep the shell stable.

Keep the world immersive.

Keep the memory alive.

Keep the camera as content.

Forward gravity only.
